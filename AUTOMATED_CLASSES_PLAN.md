# Automated Classes System Plan

This document translates your request into a practical implementation blueprint:

- Build classes simultaneously (bulk/scheduled creation)
- Child login flow fully automated
- Access restricted so students can attend only from **5:00 PM to 10:00 PM**

## 1) Product Goals

1. **Bulk class creation**
   - Create many classes at once from a schedule template or CSV import.
   - Optionally auto-create weekly recurring classes.

2. **Automated child onboarding/login**
   - Parent/admin creates child accounts once.
   - Child receives a simple secure login path (magic link / PIN / QR).
   - No manual teacher-side account setup for each class.

3. **Time-window enforcement (5 PM–10 PM)**
   - Child can join class only when local class time is between 17:00 and 22:00.
   - Outside the window, login and class-join are blocked with a clear message.

---

## 2) Suggested Architecture

## Core services

- **Auth Service**
  - Parent/admin registration
  - Child identity + session tokens
  - Short-lived class join tokens

- **Class Service**
  - Class templates, schedules, recurrence
  - Bulk creation job queue

- **Attendance Gateway**
  - Validates enrollment + time window
  - Returns `ALLOW` / `DENY` decision

- **Notification Service (optional)**
  - Sends class reminders and login links to guardians

## Data model (minimal)

- `users` (id, role: parent|child|teacher|admin)
- `children_profiles` (user_id, guardian_id, timezone)
- `classes` (id, teacher_id, title, starts_at, ends_at, timezone)
- `enrollments` (child_id, class_id, active)
- `attendance_policies` (class_id OR org_id, start_hour=17, end_hour=22)
- `join_attempt_logs` (child_id, class_id, attempted_at_utc, decision, reason)

---

## 3) Login + Join Flow

1. Guardian/admin provisions child.
2. System enrolls child into classes (manual or bulk mapping).
3. Child opens app and authenticates via simplified mechanism:
   - PIN (guardian-defined) **or**
   - Magic link to guardian-approved device **or**
   - QR code generated by guardian portal.
4. On class join click:
   - Verify enrollment.
   - Resolve class timezone.
   - Convert current UTC time to class local time.
   - Enforce allowed interval: **17:00 <= now < 22:00**.
   - If valid, issue short-lived join token and admit.

---

## 4) Time Restriction Rules (5 PM–10 PM)

Use class timezone for policy evaluation.

```text
allowed if local_time >= 17:00:00 and local_time < 22:00:00
```

### Edge-case handling

- **DST transitions:** always convert from UTC using an IANA timezone (e.g., `America/New_York`).
- **Cross-region classes:** use each class's configured timezone, not device timezone.
- **Grace periods (optional):** e.g., allow +10 minutes after 22:00 for active sessions only.

### Deny messages

- Before 5 PM: “Class access opens at 5:00 PM.”
- At/after 10 PM: “Class access closed at 10:00 PM.”

---

## 5) Bulk Class Creation (Simultaneous Build)

### Inputs

- CSV/template fields:
  - `title`, `teacher`, `timezone`, `start_time`, `duration`, `days_of_week`, `start_date`, `end_date`

### Process

1. Validate file and timezone values.
2. Generate all occurrences for date range.
3. Insert classes in batch transaction(s).
4. Enroll mapped children in batch.
5. Emit summary report (created, skipped, failed).

### Operational notes

- Use async queue for large imports.
- Ensure idempotency with import key to avoid duplicates.

---

## 6) Security & Child-Safety Basics

- Store child credentials hashed; never plaintext.
- Use short session lifetimes and refresh controls.
- Require guardian approval for new devices.
- Log all join denials and after-hours attempts.
- Provide guardian visibility dashboard (attendance + denied attempts).

## 7) API Sketch

- `POST /classes/bulk-create`
- `POST /children/{id}/enrollments/bulk`
- `POST /auth/child/login`
- `POST /classes/{id}/join`
- `GET /children/{id}/attendance`

Join response example:

```json
{
  "allowed": false,
  "reason": "OUTSIDE_ALLOWED_HOURS",
  "allowed_window": "17:00-22:00",
  "timezone": "America/New_York"
}
```
